HTML Accessibility Review

Ensuring the web is Perceivable, Operable, Understandable, and Robust for everyone.

Introduction to Accessibility

Web Content Accessibility Guidelines (WCAG): A set of guidelines for making web content more accessible to people with disabilities.

The POUR Principles:
  • Perceivable: Users must be able to perceive the information being presented.
  • Operable: Users must be able to operate the interface.
  • Understandable: Users must be able to understand the information and the operation of the interface.
  • Robust: Content must be robust enough to be interpreted reliably by a wide variety of user agents, including assistive technologies.

Assistive Technology

  • Screen readers: Software programs that read content out loud for blind or visually impaired users.
  • Large text/Braille keyboards: Hardware for users with visual impairments.
  • Screen magnifiers: Software that enlarges screen content for low-vision users.
  • Alternative pointing devices: Joysticks, trackballs, and touchpads for mobility impairments.
  • Voice recognition: Allows controlling a computer using voice commands.

Best Practices

Heading Structure

Use proper heading levels (H1-H6) to create a logical structure. This helps assistive technologies navigate the content hierarchicaly.

Tables

Use <th> for headers and <td> for data cells. Always use the <caption> element immediately after the opening <table> tag to provide context.

Form Inputs

Associate <label> elements with form inputs to clarify their purpose for screen readers.

Alt & Link Text

Use descriptive alt text for images and meaningful link text instead of "click here".

Multimedia

Provide captions and transcripts for audio/video. Audio descriptions should be provided for visual content to assist visually impaired users.

WAI-ARIA & Attributes

WAI-ARIA: Web Accessibility Initiative - Accessible Rich Internet Applications. These attributes add programmatic meaning to elements.

Key Roles

  • Document Structure: Defines relationships between sections.
  • Widget Roles: Defines interactive element functionality (e.g., scrollbars).
  • Landmark Roles: Labels primary sections (header, main, footer).
  • Live Region Roles: Announces dynamic changes (e.g., error alerts).
  • Window Roles: Defines sub-windows like modals.

Common Attributes

aria-label & aria-labelledby: Used when visual labels are missing or complex.

<button aria-label="Search">
    <i class="fas fa-search"></i>
</button>

<input type="text" aria-labelledby="search-btn">
<button type="button" id="search-btn">Search</button>

aria-hidden: Hides decorative or redundant elements from screen readers.

<button>
    <i class="fa-solid fa-gear" aria-hidden="true"></i>
    <span class="label">Settings</span>
</button>

aria-describedby: Provides additional usage context or error messages.

<form>
    <label for="password">Password:</label>
    <input type="password" id="password" aria-describedby="password-help" />
    <p id="password-help">Your password must be at least 8 characters long.</p>
</form>

Keyboard Navigation

tabindex: Makes elements focusable. Use 0 for natural order or -1 to make focusable only via script. Never use values greater than 0.

<p tabindex="-1">Sorry, there was an error with your submission.</p>

accesskey: Defines keyboard shortcuts for specific elements.

<button accesskey="s">Save</button>
<button accesskey="c">Cancel</button>
<a href="index.html" accesskey="h">Home</a>
