What Is the Purpose of WAI-ARIA, and How Does It Work?

Making static content accessible can be relatively straightforward, but dynamic content can be more challenging. This is where WAI-ARIA comes in.

Let's look at what WAI-ARIA is, its purpose, how it works, and some examples.

What is WAI-ARIA?

WAI-ARIA stands for Web Accessibility Initiative - Accessible Rich Internet Applications. It's a specification that enhances accessibility for dynamic content and UI (User Interface) components.

WCAG vs WAI-ARIA

Note: WCAG and WAI-ARIA are not the same.

  • WCAG provides general guidelines for web accessibility
  • WAI-ARIA offers specific rules for making dynamic and interactive content accessible for users of assistive technologies

Primary Purpose

The primary purpose of WAI-ARIA is to improve accessibility for dynamic content and UI components that do not have native HTML equivalents.

How WAI-ARIA Works

WAI-ARIA works by introducing a set of attributes you can add to HTML elements to provide additional semantic information. These attributes are categorized into:

  • Roles
  • States
  • Properties

1. ARIA Roles

ARIA role defines the purpose of an element within a website or web app. Here is an example of setting the role to button for a div element:

<div role="button">Click Me</div>
Click Me

By doing this you are indicating to assistive technology that the element is a button.

Important: Roles do not provide any functionality however. Merely giving this div a role of button will not make it act like a button. To make it look and behave like a button you would need to use CSS and JavaScript to get the desired result.

Example: Custom Button with HTML, CSS, and JavaScript

NOTE: Don't worry about trying to understand the CSS and JavaScript code. You will learn those languages in future modules.

HTML:

<link href="styles.css" rel="stylesheet"> <div id="custom-btn" role="button">Click Me</div> <script src="index.js"></script>

CSS (styles.css):

#custom-btn { display: inline-block; padding: 0.4em 1em; font-size: 1rem; font-family: sans-serif; color: buttontext; background-color: buttonface; border: 1px solid #ccc; border-radius: 4px; cursor: pointer; user-select: none; text-align: center; } #custom-btn:focus { outline: 2px solid Highlight; outline-offset: 2px; } #custom-btn:active { background-color: #ddd; }

JavaScript (index.js):

const button = document.getElementById("custom-btn"); button.addEventListener("click", () => { alert("Button clicked!"); }); button.addEventListener("keydown", (e) => { if (e.key === "Enter" || e.key === " ") { e.preventDefault(); button.click(); } });

Best Practice: It is always better to use the native button or input element with type="button" instead.

2. ARIA Properties

ARIA properties provide additional details about elements. For example, the aria-labelledby property lets you connect an element to a specific label:

<h2 id="header-id">About freeCodeCamp</h2> <button id="button-id" aria-labelledby="header-id button-id">Learn More</button>

This will make the elements understandable and navigable for users of assistive technologies.

Best Practices for Using WAI-ARIA

To get the best out of WAI-ARIA, follow these guidelines:

  • Use native HTML whenever possible - It generally provides more accessibility out of the box
  • Use WAI-ARIA only when HTML falls short - Don't add ARIA when native HTML elements work
  • Test with assistive technologies - Use screen readers or have people with disabilities test your work
  • Keep ARIA states and properties updated - Make sure they update with the content in real time
  • Avoid overusing ARIA - Too much ARIA can be confusing
