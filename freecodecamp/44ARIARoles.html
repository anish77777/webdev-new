What Are ARIA Roles?

ARIA stands for Accessible Rich Internet Applications.

ARIA roles specify the semantic meaning of HTML elements. They're essential for making web content accessible to people who use assistive technologies, like screen readers.

NOTE: Some of these interactive examples might include CSS and JavaScript code. Don't worry about trying to understand the code. The previews are there to help illustrate some of the main points discussed in the lesson.

HTML Semantic vs Non-Semantic

HTML has semantic and non-semantic elements, based on whether they convey meaning about their content.

Many semantic HTML elements already have an ARIA role assigned by default. For example, the button element has a default ARIA role of button.

<button>Example button</button>

But non-semantic elements don't have a role. For example, screen readers will not know how to interpret the purpose of a div if you don't specify its role explicitly.

<div> <!-- more elements go here --> </div>

The role Attribute

To specify the ARIA role of an element, you just need to add the role attribute, like this role="ARIA role".

Here is an example of using the role attribute with a value of "alert":

<div class="alert" id="exp-warning" role="alert"> <span class="hidden">Your log in session will expire in 3 minutes.</span> </div>
(Visual Alert) Your log in session will expire in 3 minutes.

It is important to note that specifying a role on an element only informs assistive technology of the purpose. It does not add any functionality or behavior.

Categories of ARIA Roles

There are six main categories of ARIA roles:

  • Document structure roles
  • Widget roles
  • Landmark roles
  • Live region roles
  • Window roles
  • Abstract roles

1. Document Structure Roles

These define the overall structure of the web page. Most are not used today because semantic HTML equivalents exist. Use roles that don't have an equivalent, such as: toolbar, tooltip, feed, math, presentation, none, and note.

Example: role="math"

<div role="math" aria-label="x squared + y squared = 3" class="math-expression"> x<sup>2</sup> + y<sup>2</sup> = 3 </div>
x2 + y2 = 3

2. Widget Roles

These define the purpose of interactive elements. Examples: scrollbar, searchbox, slider, switch, tab, tabpanel, etc.

Example: role="searchbox"

<div class="search-container" role="search"> <label for="searchbox" class="visually-hidden">Search</label> <div id="searchbox" class="searchbox" role="searchbox" aria-label="Search the site" tabindex="0" contenteditable="true"></div> <button type="button" aria-label="Submit search">Search</button> </div>

3. Landmark Roles

Landmarks help screen readers navigate sections. Examples: banner, main, navigation, search. Use these sparingly if you aren't using the equivalent semantic HTML elements (like <header> or <nav>).

Example: role="banner"

<div role="banner" class="site-banner"> <h1>Accessible Web Design</h1> <nav> <ul> <li><a href="#">Home</a></li> <li><a href="#">Articles</a></li> </ul> </nav> </div>

Accessible Web Design

  • Home
  • Articles
  • About
  • Contact

4. Live Region Roles

These define elements with dynamic content changes so screen readers can announce them. Examples: alert, status, timer.

Example: role="status"

<div class="status-demo"> <button id="update-status-btn">Check Status</button> <div id="status-msg" role="status" class="status-message">No updates yet.</div> </div>
No updates yet.

5. Window Roles

Window roles define sub-windows like modals. Examples: dialog, alertdialog. Best practice is to use the native HTML <dialog> element.

Example: role="dialog"

<button id="open-dialog">Open Dialog</button> <div id="custom-dialog" role="dialog" aria-modal="true" aria-labelledby="dialog-title" class="dialog"> <div class="dialog-content"> <h3 id="dialog-title">Confirm Action</h3> <p>Are you sure?</p> <div class="dialog-actions"> <button id="close-dialog">Cancel</button> </div> </div> </div>

Confirm Action

Are you sure you want to delete this file?

6. Abstract Roles

These help organize the document internally. They are for browser use only; developers should not use them directly.

With ARIA roles, you can create accessible and inclusive websites. They provide essential semantic information to ensure everyone has a great user experience.
